# Fast PBS Incremental — техническое задание

Версия реализации: **2.3**

## Проблема

Текущий PBS-инкремент (PXAR + chunk CDC) читает и хэширует **весь том**, даже если файлы не менялись.  
Сеть почти не используется (chunks переиспользуются), но диск и CPU заняты часами на 2+ ТБ.

SMB/FTP уже используют `fileindex` (mtime/size) и архивируют только изменённые файлы.

## Цель

Ускорить PBS-инкремент до уровня «как Veritas на файлах»: **не читать с диска неизменённые файлы**, сохраняя совместимость со снапшотами PBS.

## Не в scope v2.2

- USN Journal (v2.3)
- Блочный бэкап тома (v3.0)
- Изменение формата снапшота на PBS (остаётся PXAR + catalog)

## Архитектура

### Принцип

1. После успешного бэкапа сохранять локально:
   - **PXAR-поток** (`pbs_pxar.stream`) — точная последовательность байт, ушедших в `backup.pxar.didx`
   - **Индекс файлов** (`pbs_files.json`) — для каждого пути: size, mtime, acl_hash, диапазон байт в потоке
2. На инкременте для каждого файла:
   - `stat` + сравнение с индексом (`fileindex.NeedsContentBackup`)
   - если не менялся → вставить сохранённый PXAR-блок в поток (**без чтения файла**)
   - CDC выдаёт те же chunks → PBS переиспользует (0 upload)
3. Изменённые/новые файлы — обычный `WriteFile` (чтение с диска)

### Структуры данных

**`%ProgramData%\ClickRAX\index\<job-id>\pbs_files.json`**

```json
{
  "version": 1,
  "job_id": "...",
  "snapshot_time": "2026-07-08T12:00:00Z",
  "stream_bytes": 123456789,
  "files": {
    "folder\\file.txt": {
      "size": 1024,
      "mtime_ns": 1710000000000000000,
      "acl_hash": "abc...",
      "blob_offset": 4096,
      "blob_length": 4096
    }
  }
}
```

Ключ — путь относительно корня бэкапа, `\` как разделитель (как в SMB `fileindex`).

**`%ProgramData%\ClickRAX\index\<job-id>\pbs_entries.blob`**

- Бинарный файл с PXAR-записями отдельных файлов (concatenated blobs)
- Диапазон `[blob_offset, blob_offset + blob_length)` — полная PXAR-запись файла (FILENAME + ENTRY + PAYLOAD)
- Не дублирует весь поток снапшота: только шаблоны записей для reuse

**Расширение `pxarFilePos`** (в `backup.pxar.index.json` на PBS и локально):

- `entry_start`, `entry_end` — для restore по индексу (уже используется при записи)

### Совместимость со снапшотами

| Сценарий | Поведение |
|----------|-----------|
| Первый бэкап после обновления | Автоподготовка кэша с PBS (если есть снапшот) |
| Инкремент с кэшем | Fast path для неизменённых файлов |
| `force-full` | Удаляет `pbs_files.json` + `pbs_pxar.stream` |
| Старый снапшот без локального кэша | Fallback: полный проход (как сейчас) |
| Восстановление | Без изменений (каталог + pxar на PBS) |
| PBS `previous-index` | По-прежнему источник истины для chunk dedup |

### Первый запуск после обновления (мягкий апгрейд)

| Сценарий | Поведение |
|----------|-----------|
| Есть снапшот на PBS, локального кэша нет | Автоматически **один раз** скачивает PXAR с PBS и строит кэш (**не** полный бэкап с диска) |
| Подготовка кэша с PBS не удалась / отменена | Инкремент идёт **как в 2.1** (медленно); кэш накапливается по ходу бэкапа |
| Кэш уже есть | Быстрый инкремент сразу |
| `force-full` | Очищает кэш; нужен один обычный бэкап для накопления кэша |

**Новый полный бэкап не требуется.** Достаточно существующего снапшота на PBS.

### Безопасность fast path

Файл считается неизменённым только если:

- есть запись в `pbs_files.json`
- `size` и `mtime_ns` совпадают с `stat`
- (опционально) `acl_hash` совпадает — иначе полное чтение
- диапазон байт валиден и читается из `pbs_pxar.stream`

При сомнении — полное чтение (fallback).

### Прогресс UI

- `FilesSkipped` — число файлов, взятых из кэша
- `BytesReused` — растёт через chunk reuse (как сейчас)
- Сообщение: «Быстрый инкремент: пропущено N файлов…»

## Оценка трудозатрат

| Этап | Описание | Часы |
|------|----------|------|
| v2.2 (этот релиз) | Кэш потока + reuse в PXAR + индекс | 16–24 |
| v2.3 | USN Journal, ускорение обхода каталога | 24–40 |
| v3.0 | Блочный том (VSS + changed ranges) | 80+ |

Ожидаемый эффект v2.2 на D:\ 2.7 ТБ при <1% изменений: **минуты–часы** вместо 15–20 часов.

## Тесты

- Unit: reuse decision (mtime/size), range read
- Integration: backup → touch one file → incremental reuses rest
- Regression: force-full, no cache, first backup
